Error de gin_trgm_ops sobre array — corregido
Reemplaza search_aliases TEXT[] + intento de trgm sobre array por una columna generada concatenada y un índice trgm sobre ella:


ALTER TABLE products ADD COLUMN search_text TEXT
  GENERATED ALWAYS AS (
    lower(coalesce(name,'') || ' ' || coalesce(brand,'') || ' ' ||
          coalesce(model,'') || ' ' || array_to_string(search_aliases,' '))
  ) STORED;

CREATE INDEX ON products USING gin (search_text gin_trgm_ops);
search_aliases se queda como TEXT[] (útil para igualdad exacta con índice GIN), pero el fuzzy match va sobre search_text. Esto sí funciona.

Cambio mayor 1 — Catálogo de specs con clasificación
Lo que pides ("clasificación bien definida de cada spec con valores posibles") va más allá de extraer 5-10 columnas. Es un catálogo de tipos de spec por categoría + valores tipados por producto. Esto sustituye el JSONB suelto que había propuesto.

Tabla nueva: spec_keys (catálogo)
Cada categoría tiene N specs definidas. Cada spec tiene tipo de dato y, si es enumerable, sus valores posibles. Esto es lo que el LLM va a leer para saber qué filtros estructurados puede pedir.


CREATE TABLE spec_keys (
  id              SERIAL PRIMARY KEY,
  category_id     INT NOT NULL REFERENCES categories(id),
  name            TEXT NOT NULL,                -- "Throughput LTE"
  slug            TEXT NOT NULL,                -- "throughput_lte"
  data_type       TEXT NOT NULL,                -- 'number' | 'enum' | 'text' | 'boolean' | 'range'
  unit            TEXT,                         -- "Mbps" | "dBi" | "MHz" | NULL
  allowed_values  JSONB,                        -- enum: ["802.11ac","802.11ax"] | range: {"min":..,"max":..}
  description     TEXT,                         -- para que el extractor entienda qué es
  is_filterable   BOOLEAN DEFAULT TRUE,         -- si va al extractor de filtros
  display_order   INT DEFAULT 0,
  UNIQUE (category_id, slug)
);
Tabla nueva: product_spec_values (valor por producto)
Una fila por (producto, spec). El valor se guarda en la columna correspondiente al data_type, más el raw_text original para trazabilidad.


CREATE TABLE product_spec_values (
  product_id     BIGINT NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  spec_key_id    INT    NOT NULL REFERENCES spec_keys(id),
  value_number   NUMERIC,
  value_text     TEXT,
  value_enum     TEXT,                          -- debe estar en allowed_values del spec_key
  value_boolean  BOOLEAN,
  raw_text       TEXT NOT NULL,                 -- valor original sin parsear, para auditoría
  PRIMARY KEY (product_id, spec_key_id)
);

CREATE INDEX ON product_spec_values (spec_key_id, value_number);
CREATE INDEX ON product_spec_values (spec_key_id, value_enum);
Cómo se construye el catálogo desde lo que ya tienes
product_specs.specs (JSONB) sigue existiendo como fuente cruda. Encima va un proceso de normalización en dos pasos:

Discovery (una vez por categoría): un script con LLM lee todas las specs de los productos de la categoría, agrupa claves equivalentes ("Velocidad LTE" / "Throughput LTE" / "LTE Speed" → un solo spec_key), infiere data_type, unit, y allowed_values para enums.
Mapping (por producto): otro paso parsea el valor de cada spec y lo guarda en product_spec_values. Si no parsea (ej. valor ambiguo), va a value_text + raw_text y se loguea para revisión manual.
Esto es trabajo de ETL real (1–2 días) pero es exactamente lo que habilita el prefiltrado preciso por specs numéricas y enumerables. Sin esto, "router con throughput >1Gbps" no funciona.

Trade-off honesto
JSONB suelto seguiría funcionando con menos esquema. Pero dijiste que quieres que el LLM pueda prefiltrar por specs estructuradas — eso requiere catálogo. Esta es la decisión correcta para tu objetivo, asumiendo el costo del ETL de normalización.

Cambio mayor 2 — Tipos de información (info_types)
Lo que pediste con "atributo de tipo para saber si... solo me piden información del producto, descripción, especificaciones, categorías, características, relaciones, filtros" es el patrón de router de intents. El extractor NLU no solo extrae filtros: clasifica qué tipos de información hay que recuperar y de qué fuente.

Inventario de fuentes y cómo se resuelve cada una
Tipo de info	Fuente	Mecanismo

Qué hacer en realidad
Una sola columna JSONB normalizada por LLM en el ETL. Sin catálogo separado.


ALTER TABLE product_specs
  ADD COLUMN specs_normalized JSONB DEFAULT '{}';
El ETL hace una pasada por producto:

"Aquí están las specs crudas de un producto categoría=X. Devuélveme un JSON con claves snake_case, valores numéricos donde tenga sentido, unidades en formato estándar (Mbps, dBi, MHz, V, W). Estas son las claves ya usadas en esta categoría: [lista]. Reusa esas claves cuando apliquen."

Salida típica:


{
  "throughput_lte_mbps": 150,
  "wifi_standard": "802.11ac",

  "ports_serial": 1,
  "voltage_v": 12,
  "operating_temp_min_c": -40,
  "operating_temp_max_c": 75
}
Eso se guarda. Punto.

Cómo funciona cada cosa con esto
Filtros numéricos en query ("router con throughput >1Gbps"):


SELECT p.* FROM products p
JOIN product_specs ps ON ps.product_id = p.id
WHERE p.category_id = $1
  AND (ps.specs_normalized->>'throughput_lte_mbps')::numeric >= 1000;
Sobre <100 productos por categoría, sin índice, corre en <10ms.

Extractor LLM en query time: antes de extraer filtros, hace una sola query para saber qué claves existen para la categoría candidata:


SELECT DISTINCT jsonb_object_keys(specs_normalized) AS key
FROM product_specs ps
JOIN products p ON p.id = ps.product_id
WHERE p.category_id = $1;
Le pasas eso al LLM como "claves disponibles" y él decide cuáles filtrar. Este es el trick que reemplaza al catálogo spec_keys: el catálogo emerge del uso, no se cura a mano.

Inconsistencia de claves entre productos: sí va a pasar (un router dice speed_mbps, otro throughput_mbps). Mitigación pragmática: el ETL ve la lista de claves ya usadas en la categoría y reusa. Si quieres limpieza adicional, una query mensual te muestra claves con <3 productos = candidatas a fusionar. Ese es el momento correcto para curar, no antes.

Índices: ninguno al inicio. Si una clave se vuelve hot (la usas en muchas queries), agregas un índice de expresión:


CREATE INDEX ON product_specs (((specs_normalized->>'throughput_lte_mbps')::numeric));
Trabajo: 1 línea cuando la query se vuelva lenta. No antes.

Prompt para specs_normalized (paso 9)
Este es el único paso de IA en el ETL de catálogo. Se ejecuta por producto. Recibe specs crudas + claves ya vistas en su categoría. Devuelve JSON plano.


ROL
Eres un normalizador de specs técnicas de productos B2B (telecom, networking,
IoT industrial). Tu salida alimenta una base de datos para RAG.

ENTRADA
{
  "category_slug": "modems-y-routers",
  "category_name": "Módems y routers",
  "product_name": "Robustel EG5100",
  "specs_raw": [
    {"name": "Throughput LTE", "value": "150 Mbps DL / 50 Mbps UL", "section": "Conectividad"},
    {"name": "Puertos LAN", "value": "4 x RJ45 10/100", "section": "Interfaces"},
    ...
  ],
  "known_keys_in_category": [
    "throughput_lte_mbps", "throughput_lte_ul_mbps", "ports_lan", "wifi_standard",
    "voltage_v", "operating_temp_min_c", "operating_temp_max_c"
  ]
}

REGLAS
1. Devuelve un objeto JSON PLANO (sin anidamiento). Cada clave es snake_case.
2. REUSA claves de "known_keys_in_category" siempre que la spec encaje semánticamente.
   Solo crea una clave nueva si ninguna existente aplica.
3. Valores numéricos van como NUMBER (no string). Convierte unidades a la
   estándar implícita en el nombre de la clave:
     *_mbps -> Mbps,  *_v -> Volts,  *_c -> Celsius,  *_dbi -> dBi,
     *_mhz -> MHz,    *_mm -> mm,    *_g -> gramos,   *_w -> Watts.
4. Si una spec tiene rango (ej. "-40 a 75 °C"), divídela en dos claves:
   "<base>_min_<unit>" y "<base>_max_<unit>".
5. Si una spec tiene múltiples valores discretos (ej. "802.11b/g/n/ac"),
   guárdala como ARRAY de strings normalizados: ["802.11b","802.11g","802.11n","802.11ac"].
6. Booleanos para Si/No, presencia/ausencia: true/false.
7. Si no puedes parsear un valor con confianza, NO lo inventes: omítelo.
   Lo que no normalices queda en specs (crudo) y no se pierde.
8. NO incluyas claves con valor null, "", [] o "N/A".
9. NO incluyas claves de marca, nombre, modelo, descripción ni URLs:
   esas viven en otras columnas.
10. Salida = SOLO el objeto JSON. Sin explicaciones, sin markdown, sin código.

EJEMPLO DE SALIDA
{
  "throughput_lte_dl_mbps": 150,
  "throughput_lte_ul_mbps": 50,
  "ports_lan": 4,
  "ports_lan_speed_mbps": 100,
  "voltage_v": 12,
  "operating_temp_min_c": -40,
  "operating_temp_max_c": 75,
  "wifi_standard": ["802.11b","802.11g","802.11n"],
  "has_serial": true

Cómo construir known_keys_in_category en cada llamada:


SELECT array_agg(DISTINCT key ORDER BY key)
FROM product_specs ps
JOIN products p ON p.id = ps.product_id
CROSS JOIN LATERAL jsonb_object_keys(ps.specs_normalized) AS key
WHERE p.category_id = $1;
La primera ejecución de la categoría devuelve [] y el LLM crea el vocabulario. A partir del segundo producto, ya hay base de reuso. Las claves convergen rápido — para 10–15 productos por categoría, el vocabulario se estabiliza.

Modelo recomendado: claude-sonnet-4-6 o gpt-4o-mini. No necesitas opus para esto. Costo total para los 67 productos: <$1.

Validación post-normalización: una query mensual identifica claves "huérfanas" (con <3 productos), candidatas a fusionar:


SELECT key, COUNT(*) AS products_with_key
FROM product_specs ps
CROSS JOIN LATERAL jsonb_object_keys(ps.specs_normalized) AS key
GROUP BY key
HAVING COUNT(*) < 3
ORDER BY products_with_key, key;

###  Multi-term Si/No — comportamiento esperado, no corregir

Tres productos tienen valores opuestos para el mismo atributo:

```
robustel-r2011    -> pa_wifi -> ['no', 'si']
robustel-r1510-4l -> pa_wifi -> ['no', 'si']
robustel-r2110    -> pa_wifi -> ['no', 'si']
suntech-kit-de-voz -> pa_audio-en-cabina -> ['no', 'si']
```

**Decisión**: no filtrar en ETL. El sitio web de origen muestra el mismo
comportamiento (el producto aparece tanto en el filtro "Con WiFi" como en
"Sin WiFi" porque tiene variantes). Replicar eso en la DB es lo correcto
para mantener consistencia. El schema lo soporta con la PK compuesto de
`product_attribute_values`.

Multi-term legítimo (no es problema):
- `pa_uso -> ['empresarial', 'industrial']`: el producto sirve ambos casos
- `pa_red-celular -> ['3g-4g', '5g']`: el producto soporta ambas redes
- `pa_red -> ['2g', '4g-lte-m-nb-iot']`: dual-band


### Diseño que llegaría desde cero

Idéntico en lo grande a `schema.sql`. Las decisiones de fondo coinciden:

1. **Postgres + pgvector + pg_trgm** en una DB. Para 74 SKUs, cualquier
   otra cosa es over-engineering.
2. **EAV controlado para los `pa_*`**. Multi-term existe (29 productos),
   atributos varían por categoría (`pa_dbi` sólo aplica a antenas), opciones
   son discretas y reutilizables. Una columna por atributo en `products`
   sería un infierno.
3. **Specs en JSONB doble**: crudo (`specs`) preserva la fuente con
   `section` y `items`; normalizado (`specs_normalized`) por LLM permite
   queries numéricas (`>=`, `<=`) sin rigidez de schema. A este volumen un
   catálogo formal de spec_keys es overkill.
4. **Software con tabla propia**. Con 7 software canónicos contra 38
   productos que los referencian, la deduplicación ahorra chunks y mantiene
   coherencia (un mismo `RobustOS` no se describe distinto cada vez).
5. **rag_chunks con metadata denormalizada**. El filtrado del 80 % de las
   queries va a ser por `category_id`, `is_new` y `attribute_slugs`. Si
   eso requiere JOINs, mata la latencia.
6. **Relaciones dirigidas con `relation_type`**. Hoy todas son
   `recommended_product` con `weight=0.7` (el peso no aporta señal aún),
   pero la tabla queda lista para `replaces_product` y `similar_specs`.

ALTER TABLE attribute_options
