Estrategia para filtros dinámicos
Para responder "qué filtros le muestro al usuario para la categoría X":


SELECT a.id, a.name, a.taxonomy, a.type,
       jsonb_agg(jsonb_build_object('id', ao.id, 'name', ao.name, 'slug', ao.slug)
                 ORDER BY ao.name) AS options
FROM category_attributes ca
JOIN attributes a            ON a.id  = ca.attribute_id
JOIN attribute_options ao    ON ao.attribute_id = a.id
WHERE ca.category_id = $1
GROUP BY a.id
ORDER BY ca.display_order;
Para filtrar productos por una combinación de filtros (lógica AND entre atributos, OR entre opciones del mismo atributo):


SELECT p.*
FROM products p
WHERE p.category_id = $1
  AND p.is_new = COALESCE($2, p.is_new)
  AND NOT EXISTS (
    SELECT 1 FROM unnest($3::int[]) AS attr_id
    WHERE NOT EXISTS (
      SELECT 1 FROM product_attribute_values pav
      JOIN unnest($4::int[]) AS option_id ON pav.attribute_option_id = option_id
      WHERE pav.product_id = p.id AND pav.attribute_id = attr_id
    )
  );
(Patrón "all attributes must match at least one of the requested options".)

8. Estrategia para specs / descripciones / software
specs estructuradas → JSONB en product_specs.specs (consultable con @>, jsonb_path_query, etc).
specs_text / features_text → texto markdown; va a embeddings (chunk_type specs / features).
description corta → products.description; va como chunk tipo description.
Software canónico → tabla software; un solo embedding (chunk_type software) referenciable desde múltiples productos vía products.software_id. Cuando el RAG retorna ese chunk, en el query expansion se pueden traer todos los productos que lo comparten.
Esto resuelve el problema de "12 chunks idénticos del mismo software ensuciando el top-k".

9. Estrategia para relaciones / recomendaciones
Tres origenes de recomendación posibles:

Explícitas (product_relations tipo recommended_product) — del catálogo Woo.
Replacement (tipo replaces_product) — extraídas del texto por el nodo de relaciones.
Inferidas por similarity — opcionalmente, calcular cosine de embeddings entre productos de la misma categoría y materializar como relation_type='similar_specs' con source='inferred'.
"Productos más recomendados en una categoría":


SELECT p.id, p.name, COUNT(*) AS times_recommended
FROM product_relations pr
JOIN products p ON p.id = pr.target_product_id
WHERE p.category_id = $1
  AND pr.relation_type = 'recommended_product'
GROUP BY p.id
ORDER BY times_recommended DESC
LIMIT 10;
10. Estrategia para RAG y prefiltrado de embeddings
Patrón filter-then-rank:

Parser de la pregunta del usuario (NLU ligero o LLM con función) → intent + filters:
category_slug (si menciona categoría)
is_new (si dice "nuevos")
brand (si menciona marca)
required_attributes (combinación de filtros)
Query a Postgres con prefiltrado sobre la metadata denormalizada de rag_chunks, ordenando por distancia:

SELECT c.product_id, c.chunk_type, c.content,
       1 - (c.embedding <=> $1::vector) AS similarity
FROM rag_chunks c
WHERE c.category_slug = COALESCE($2, c.category_slug)
  AND c.is_new        = COALESCE($3, c.is_new)
  AND ($4::text[] IS NULL OR c.attribute_slugs && $4)
ORDER BY c.embedding <=> $1::vector
LIMIT 20;
Deduplicación post-retrieval por software_group y por product_id (no devolver 5 chunks del mismo producto).
Si el filtrado deja menos de N resultados, hacer fallback a una búsqueda más laxa (sin atributos, solo categoría).
Reglas de inclusión/exclusión (para responder "qué entra y qué se excluye del retrieval"):

Excluir chunks de productos sin specs si la pregunta es técnica.
Excluir software no canónico (siempre devolver el canónico).
Si el usuario pidió "nuevos", is_new=TRUE es duro (no fallback).

SELECT p.* FROM products p
WHERE p.category_id = (SELECT category_id FROM products WHERE slug=$1)
  AND p.id <> (SELECT id FROM products WHERE slug=$1)
ORDER BY (
  SELECT AVG(embedding <=> (
    SELECT embedding FROM rag_chunks WHERE product_id=p.id AND chunk_type='specs' LIMIT 1
  )) FROM rag_chunks WHERE product_id=(SELECT id FROM products WHERE slug=$1)
) ASC
LIMIT 5;

¿Cuál es el caso de uso?

LLM-driven Q&A sobre el catálogo. Las preguntas mezclan:
- **Filtros estructurados duros** ("routers nuevos", "5G", "industrial"):
  responder por SQL puro, no por similarity.
- **Información narrativa** ("¿qué hace este router?", "compárame X con Y"):
  responder por embeddings + LLM.
- **Numéricos** ("throughput > 1 Gbps"): responder por SQL sobre JSON normalizado.
- **Recomendaciones** ("¿qué se recomienda con X?"): responder por SQL sobre tabla de relaciones.

A 74 productos no hay problema de escala. El problema es de **precisión**:
el LLM no puede inventar specs. Hay que prefiltrar para que sólo vea
contexto relevante.
